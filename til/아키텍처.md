# 아키텍처

- 시스템의 전체 구조 + 중요한 설계 결정들의 집합
- 시스템 전체 방향을 결정하는 최상위 설계

1. 구조 (Structure)
2. 설계 (Design)
3. 프레임워크 (Framework)
4. 청사진 (Blueprint)
## 1.구조(Structure)
- 시스템을 이루는 구성 요소들의 배치와 관계
- 구조를 나누는 이유는 유지보수와 확장성을 높이기 위해서
#### 전체 구조
- 클라이언트 - 서버 - 데이터베이스

#### 서버 내부 구조
- Controller
    - 클라이언트의 요청을 처음 받는 계층
- Service
    - 비즈니스 로직(프로그램의 핵심 규칙이나 처리 과정)을 처리하는 계층
- Repository
    - 데이터베이스와 직접 연결되는 계층

#### 예시(로그인 기능)
- 클라이언트 → Controller → Service → Repository → 데이터베이스

## 2.설계(Design)
- 구조를 기반으로 한 구체적인 구현 방식
- 코드의 효율성과 가독성을 높이기 위해 필요
### 포함 요소
- 클래스 설계
- API 설계
- 데이터 흐름
- 알고리즘

#### 예시
- SOLID 원칙 적용
- DTO 설계
- REST API(클라이언트와 서버가 약속된 방식으로 대화하는 규칙) 설계

## 3. 프레임워크(Framework)
- 설계를 쉽게 구현하도록 도와주는 규칙 + 도구의 집합
- 프레임워크를 사용하면 개발 시간을 줄이고 일정한 구조를 유지할 수 있습니다.

#### 특징
- 개발 방식이 정해져 있음
- 재사용 가능한 구조 제공
- 생산성 증가

#### 예시
- Spring-스프링(Java 기반 백엔드 프레임워크)
- Django-장고(Python으로 만든 백엔드 웹 프레임워크)
- React-리액트(JavaScript 기반 프론트엔드 라이브러리/프레임워크처럼 쓰이는 도구)

## 4. 청사진(Blueprint)
- 아키텍처를 시각적으로 표현한 설계도
- 청사진은 개발 전에 구조를 미리 확인하고 팀원들과 공유하기 위해 사용됩니다.
- 만약 청사진 없이 개발하면 구조가 꼬일 수 있다.
### 특징
- 시스템을 한눈에 이해 가능
- 팀 간 소통 도구
- 개발 기준 제공

#### 예시
- 아키텍처 다이어그램(시스템의 전체 구조를 그림으로 나타낸 것)
- UML(프로그램 구조를 그림으로 표현하는 방법)
- 시스템 구성도
