## HTTP

- 웹에서 클라이언트와 서버가 데이터를 주고받기 위한 프로토콜
- http:// 로 시작
- 데이터를 암호화하지 않고 평문으로 전송

---

#### 동작 방식
- 사용자가 웹사이트 접속 요청
- 서버가 요청을 처리하고 응답 반환
- 요청(Request) - 응답(Response) 구조

---

#### 무상태성 (Stateless)
- 서버가 이전 요청을 기억하지 않음
- 매 요청마다 필요한 정보를 포함해야 함

---

#### 장점
- 속도가 빠름
- 구조가 단순하여 구현이 쉬움
- 리소스 사용이 적음

#### 단점
- 데이터가 암호화되지 않음
- 중간에 데이터 탈취(해킹) 가능
- 보안에 매우 취약

---

## HTTPS

- HTTP에 보안 기능(SSL/TLS)이 추가된 프로토콜
- https:// 로 시작
- 데이터를 암호화하여 전송

---

#### 동작 방식
- SSL/TLS를 이용해 암호화 통신
- 서버 인증서로 신뢰성 검증
- 안전한 연결 후 데이터 전송

---

#### 장점
- 데이터 암호화 (도청 방지)
- 데이터 무결성 보장 (변조 방지)
- 서버 인증 가능 (가짜 사이트 방지)
- SEO 및 최신 웹 기능 지원

#### 단점
- HTTP보다 속도가 약간 느림
- 인증서 발급 및 설정 필요
- 서버 자원 사용 증가

---

## HTTP 상태 코드

- 서버가 요청 처리 결과를 숫자로 반환

---

#### 1xx (정보)
- 요청 처리 중

#### 2xx (성공)
- 200: 요청 성공
- 201: 생성 성공

#### 3xx (리다이렉션)
- 301: 영구 이동
- 302: 임시 이동

#### 4xx (클라이언트 오류)
- 400: 잘못된 요청
- 401: 인증 필요
- 403: 접근 금지
- 404: 페이지 없음

#### 5xx (서버 오류)
- 500: 서버 내부 오류
- 503: 서비스 불가

---

## HTTP vs HTTPS 핵심 차이

| 구분 | HTTP | HTTPS |
|------|------|-------|
| 보안 | 없음 | 있음 (암호화) |
| 속도 | 빠름 | 약간 느림 |
| 데이터 | 평문 전송 | 암호화 전송 |
| 사용 예 | 단순 정보 | 로그인, 결제 |

---

## PUT vs PATCH 차이

#### PUT
- 전체 데이터 수정
- 기존 데이터를 모두 대체

#### PATCH
- 일부 데이터만 수정
- 필요한 부분만 변경

---

#### 차이 정리

| 구분 | PUT | PATCH |
|------|-----|-------|
| 수정 범위 | 전체 | 일부 |
| 데이터 | 전체 필요 | 일부만 필요 |

---

## 한 줄 정리
- HTTP: 빠르지만 위험
- HTTPS: 조금 느리지만 안전
- Stateless: 상태를 기억하지 않음
- 상태 코드: 요청 결과를 숫자로 표현
- PUT: 전체 수정 / PATCH: 부분 수정
